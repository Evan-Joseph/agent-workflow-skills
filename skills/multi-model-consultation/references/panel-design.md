# Panel Design

Use this reference when deciding how many models to ask and which kind of panel to use.

## Start small

Most tasks do not need six models.

Use:

- **2 models** when you want a quick cross-check
- **3 models** when you expect one disagreement and want a tie-break shape
- **4 to 6 models** only for harder design, planning, coding, or arbitration tasks

Larger panels cost more and often add repetition.

## Panel shapes

### Analysis

Use for:

- product and system tradeoffs
- planning
- proposal review
- decision support

Look for:

- whether the models frame the problem the same way
- whether any of them identify a missing constraint

### Coding

Use for:

- implementation review
- debugging hypotheses
- architecture choices
- code generation comparison

Look for:

- agreement on root cause
- concrete implementation differences
- hidden migration or maintenance costs

### Writing

Use for:

- naming
- copy direction
- structure and tone

Look for:

- whether the answers are actually distinct
- whether any answer sounds polished but dodges the brief

### Vision

Use for:

- screenshot critique
- layout comparison
- image prompt review
- OCR or UI interpretation cross-checks

Look for:

- object detection mismatches
- differences in reading hierarchy
- text extraction disagreements

### Premium

Use only when:

- the user asked for it
- a lower-cost panel still leaves a meaningful unresolved decision
- the task is expensive enough that another hour of rework would cost more than the extra inference

## Arbitration

If one expensive model is used only as an arbiter, shorten the packet.

Send:

- the goal
- the key facts
- a compact summary of the disagreements
- the exact arbitration question

Do not dump full raw transcripts unless the disagreement depends on exact wording.
